staging area (index)

el area de control de cambios, corresponde a un archivo que se encuentra en la carpeta .git, y se utiliza para registrar o realizar las cap de pantallas del codigo, tambien conocidas como snapshot de los diferentes archivos del proyecto. Aqui se realiza el seguimiento de los mismos, para realizar la confirmacion y poder acceder a los diferentes cambios dentro de la historia de nuestro proyecto. Es importante tener en ciuenta que todo archivo que no sea agregado al index no se podra recuperar en caso de ser eliminado.

  • git status muestra el estado de los archivos con respecto a la ultima confirmacion, en caso que exista.
    • untracked: al archivo no se le esta realizando seguimiento
    • staged: los cambios fueron agregados pero aun no estan confirmados
    • modified: el archivo difiere con respecto a lo almacenado en la ultima confirmacion
    • commited: los cambios fueron almacenados en una confirmacion
  • git add <file agrega al INDEX el/los archivos seleccionados
  • git rm --cached <file> quita del INDEX el archivo seleccionado sin modificar el original, cuando todavia no fue confirmado
  • git restored --staged <file> remueve los cambios de los archivos agregados al index

Repository

Area donde se almacenan los cambios, mediante confirmaciones, en formato BLOB (Binary Large Object File). Solamente se confirman las modificaciones agregadas al INDEX. Y mediante apuntadores; podemos navegar dentro del historial.

  • git commit abre el editor para realizar la confirmacion.
  • git blame muestra linea por linea los autores de las confirmaciones, en caso que existan.
  • git log muestra el historial de confirmaciones en orden descendente
  • git remote add <alias> <url> agrega una direccion remota asignandole un alias
  • git fetch <remote> obtiene el historial de cambios de un repositorio remoto
  • git push <remote> <url envia a un repositorio remoto los cambios
